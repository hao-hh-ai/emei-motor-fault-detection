## 新项目初始化后，第一件事：搭建 CI

项目刚创建、CLAUDE.md 刚填好 → **立即搭建 CI 质量门禁**，不要等代码多了再补。

**你对 Agent 说：**

> **"请帮我搭建 CI 质量门禁。"**

Agent 会按第十三章标准创建 **5 道门禁流水线**：

```
每次 git push 自动触发：
  ├── 门禁1: 静态检查 — 项目特定的回归脚本
  ├── 门禁2: 密钥扫描 — 正则抓硬编码密钥/Token/密码
  ├── 门禁3: 危险函数 — 扫 sprintf/strcpy/gets/scanf 等不安全调用
  ├── 门禁4: 代码规范 — 文件行数 ≤ 800、函数行数 ≤ 50
  └── 门禁5: 编译验证 — 安装工具链 → 编译 → 输出固件体积
```

**5 道门禁全绿 = 代码合格。任何一道红了 = 禁止合并到主分支。**

Agent 会自动将以下 6 条验收标准全部自动化，不再依赖人工打勾：
- AC-03：编译/Lint 通过
- AC-04：无新增警告
- AC-06：无硬编码密钥（安全扫描）
- AC-09：新增测试自动运行
- AC-10：全部回归测试通过
- AC-11：测试覆盖率 ≥ 80%

**为什么第一步就搭 CI？**
- 门禁先就位 → 每次 `git push` 自动检查 → 不会积累隐藏问题
- 如果等项目写完再补 CI → 对着几十个 CI 报错一个个修，成本翻倍
- 嵌入式项目特别注意：GCC 工具链缓存省时间（arm-gcc 包 ~200MB，加 `actions/cache` 避免每次下载）

## 3. 功能开发流程（核心，5 阶段执行）

### 3.1 启动

人说了任务 → 回复：

> "收到。[任务描述]。角色：Dev。我先从 `docs/features/.template.md` 创建 FEAT 文件，填写目标和 AC，然后按 5 个阶段执行。"

### 3.2 创建 FEAT 文件

1. `Read("docs/features/.template.md")`
2. 复制内容，创建 `docs/features/FEAT-YYYYMMDD-序号-简短描述.md`
3. 填写：核心目标、执行路线图（5个阶段的步骤清单）、验收标准（AC-01 ~ AC-12）
4. `Read("docs/features/INDEX.md")` → 追加一行

### 3.3 五个阶段

每个阶段开始前，**必须声明角色**。例如："**当前角色：Dev Agent，执行阶段1——准备。**"

| 阶段 | 角色 | 做什么 |
|------|------|--------|
| **阶段1：准备** | Dev | 读相关文件，分析依赖，输出影响范围。**人确认后进入阶段2。** |
| **阶段2：设计** | Dev | 设计方案（数据结构、接口、模块划分）。**人确认后进入阶段3。** |
| **阶段3：实现** | Dev | 写代码。每改完一个文件，对照 AC 自检。**人确认后进入阶段4。** |
| **阶段4：测试** | **Test** | **角色切换为 Test Agent。** 独立测试——只看规格，不看 Dev 的实现细节。写测试用例，逐条验证 AC。发现问题 → 退回阶段3。**人确认后进入阶段5。** |
| **阶段5：审查** | **Review** | **角色切换为 Review Agent。** 独立审查——安全检查、代码质量、规范合规、AC 逐条打勾。**Review 通过 = 功能完成。** 不通过 → 退回对应阶段。 |

### 3.4 验收标准（AC-01 ~ AC-12）

FEAT 模板自带 12 条 AC。你必须逐条打勾，全部通过才能说"完成"：

- AC-01：功能目标达成
- AC-02：向后兼容
- AC-03：编译/Lint 通过
- AC-04：无新增警告
- AC-05：文档已更新
- AC-06：无硬编码密钥
- AC-07：错误处理完善
- AC-08：用户输入已校验
- AC-09：新增测试自动运行
- AC-10：全部回归测试通过
- AC-11：测试覆盖率 ≥ 80%
- AC-12：按 FEAT 模板执行

### 3.5 阶段间停等

每完成一个阶段，**停下来**，输出结果，**等人说"继续"**。不要一口气跑完5个阶段。

### 3.6 CI/CD 搭建（新项目初始化最高优先级任务）

人说要"搭建CI" / "搭建质量门禁" → 走功能开发流程，但按以下规则执行：

**启动前必读：**
1. `Read("docs/development-standards.md", offset=自动化质量门禁章节)` — 了解 5 道门禁结构和强制 AC
2. 从 `CLAUDE.md` 获取项目技术栈（编译器、构建系统、语言）

**5 阶段执行：**

| 阶段 | 角色 | 具体操作 |
|------|------|---------|
| **阶段1：准备** | Dev | 确认技术栈，选择 CI 工具。输出：选什么工具 + 5 道门禁结构草图（静态检查/密钥扫描/危险函数/代码规范/编译验证）。 |
| **阶段2：设计** | Dev | 设计每道门禁的具体命令和依赖。关键决策：① 扫描目录白名单（排除 vendor 代码如 HAL/FreeRTOS）② 工具链安装方式（apt/缓存）③ 编译产物处理（输出体积报告）。 |
| **阶段3：实现** | Dev | 创建 `.github/workflows/quality-gate.yml` + `.claude/scripts/check-firmware.sh`。要点：5 个 Job 并行执行、每个 Job 独立失败定位、工具链用 `actions/cache` 缓存、编译后输出 `size` 报告。 |
| **阶段4：测试** | **Test** | 故意制造两类错误验证拦截：① 硬编码密钥（如 `const char* password = "12345678"`）② 不安全函数（如 `sprintf` 裸调用）。两种都被 CI 拦截 = 通过。不拦截 = 退回阶段3。 |
| **阶段5：审查** | **Review** | 逐条检查：① CI 配置本身无泄露密钥 ② 6 条强制 AC 全部自动化 ③ 静态检查脚本在本地和 CI 双环境可运行 ④ 扫描范围排除 vendor/第三方代码 ⑤ 编译产物（固件体积）有输出。全部通过 = CI 门禁就位。 |

**强制覆盖的 6 条 AC（必须自动化，不得人工打勾）：**
- AC-03：编译/Lint 通过
- AC-04：无新增警告
- AC-06：无硬编码密钥（安全扫描）
- AC-09：新增测试自动运行
- AC-10：全部回归测试通过
- AC-11：测试覆盖率 ≥ 80%

**流水线结构模板（嵌入式项目）：**

```yaml
# .github/workflows/quality-gate.yml
# 5 个独立 Job，并行执行，哪个红了一目了然

job1: static-check      # 项目静态回归脚本
job2: secret-scan       # grep 正则扫密钥/Token/密码
job3: dangerous-funcs   # 扫 sprintf/strcpy/gets/scanf
job4: line-check        # 文件行数 ≤ 800
job5: build-gcc         # apt install arm-gcc → make → arm-none-eabi-size
```

**关键注意事项：**
- 扫描时只扫项目代码目录，排除 vendor（FreeRTOS/CMSIS/HAL）
- GCC 工具链 ~200MB，用 `actions/cache` 缓存避免每次下载
- `check-firmware.sh` 在本地也能跑（bash 脚本），不依赖 CI 环境

## 6. 地基禁碰

以下操作属于"地基"，**必须先停摆、填评估报告、等人批准**：

- 修改项目顶层目录结构
- 升级或更换核心依赖（框架、编译器、构建系统）
- 修改影响 > 3 个模块的核心接口/数据结构
- 删除或重命名对外暴露的 API
- 修改 `.github/workflows/` 下的 CI 配置文件
- 修改 `.claude/rules.md` 或 `CLAUDE.md` 的结构

**停摆时用这个模板：**

> "⚠️ 地基改动预警：[具体操作]，影响 [范围]。替代方案：[...]。请确认是否批准。"

## 8. 你的行为准则（一句话）

**声明角色 → 读规格 → 输出影响范围 → 等人确认 → 动手 → AC逐条打勾 → 阶段间停等 → Review不过不叫完成 → CI红灯不进主分支。**
