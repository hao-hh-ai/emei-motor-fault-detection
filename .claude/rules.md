## 1. 理解优先，切勿跳过分析

- 收到任务后，先读相关文件、梳依赖链、输出影响范围，等用户确认后再写代码。
- 不猜测需求，不创造不存在的接口，不修改无关代码。
- **Never write code first. → Understand → Plan → Implement.**

### 1.1 工具链迁移 / ESP-IDF 升级 → 必须做配置源对比

触发条件：ESP-IDF 版本升级、工具链切换、sdkconfig 重新生成。
**强制动作：** 对比 `sdkconfig` 中的配置项 vs 代码中实际依赖的 Kconfig 选项，差异项列入"迁移后需手动恢复"。

> ESP-IDF 升级/重配置可能改变 Kconfig 默认值，导致外设引脚/缓冲区大小等参数静默变化。

### 1.2 工具链切换后功能异常 → 先查工具链差异，再查代码

**核心原则：唯一变量是编译器行为，先排除工具链差异。**
排查优先级：sdkconfig 配置 → 链接脚本(堆/栈) → 预定义宏 → 最后才怀疑代码。

## 2. 最小改动原则

- 只改目标问题所需的代码。
- 不重构无关代码，不删除现有功能。
- 保持向后兼容。
- **如果你发现一个"顺手修"的东西 — 停下来上报，不要顺手。**

## 3. 安全红线（违反即退回）

- 严禁硬编码密钥/Token/密码到代码中。WiFi 密码等敏感配置通过 Kconfig (sdkconfig) 管理。
- 所有外部输入（MQTT 消息/串口数据）必须校验后才使用。
- 敏感日志脱敏：密码、Token 打印为 `***`。
- C/C++：数组不越界、缓冲区不加宽、格式化不注入。

## 4. 改动前输出影响范围

无论改动大小，计划中必须包含：

```
📋 影响范围
├── 修改文件: xxx
├── 新增文件: 无 / xxx
├── 接口变化: 无 / xxx() 签名变化
├── 存储变化: 无 / 配置/DB 结构变化
├── 兼容性: ✅ 完全兼容 / ⚠️ 需迁移 / ❌ 不兼容
└── 风险: 低 / 中 / 高
```

## 5. 验证后再说"完成"

- 编译通过、回归检查通过，才声称"已修复/已完成"。
- 不通过的贴错误信息，不要隐瞒。

## 6. Bug修复后必须记录

- 每修完一个Bug，必须在 `docs/bugs/` 下创建 `BUG-YYYYMMDD-序号-简短描述.md` 文件，并更新 `INDEX.md` 汇总表。
- 记录内容：Bug ID、标题、等级、根因（一句话）、修复方案（一句话）、影响文件。
- 模板见 `docs/bugs/.template.md`，详细格式见 `docs/development-standards.md` → Bug记录规范章节。

## 7. 善用项目工具

- 有 CodeGraph → 查代码优先用 `codegraph explore`
- 有 CLAUDE.md → 查项目概览和规范索引

## 8. 架构/地基改动 — 禁碰即停（硬闸门）

### 8.1 什么是"地基"（触发条件）
以下任何一项变更均视为"动地基"，**严禁 AI 擅自实施**：
- 目录结构重构
- 核心依赖更换（ESP-IDF 版本、FreeRTOS → 其他 RTOS）
- 全局接口/协议变更（MQTT 消息格式、引脚定义重排）
- 编译/构建系统更改（CMake → Makefile）
- sdkconfig 关键配置变更（Flash 大小、CPU 频率等）

### 8.2 遇到"地基"时的强制动作
1. 紧急停摆，输出"⚠️ 地基触碰警告"
2. 生成地基影响评估报告（含：触发类型、当前任务、改动内容、影响范围、替代方案2个、风险评估、建议策略）
3. 等待用户明确批准（"批准"或"同意"）后才可编码

### 8.3 例外情况
- 用户已在 FEAT-*.md 中明确写明了"需修改地基"
- 新增完全独立的模块文件

## 9. CI 门禁铁律 — 红灯不进主分支

- CI 流水线（`quality-gate.yml`）任何一道门禁未通过，代码不得合并到主分支。
- 本地提交前必须运行项目静态检查脚本。跳过检查直接 push 视为违规。
- CI 配置文件（`.github/workflows/`）属于"地基"（见第 8 条），修改必须走地基评估流程。

## 10. 强制角色分离 — 不准自审自判

### 9.1 角色定义
| 角色 | 职责 | 权限 |
|------|------|------|
| Dev Agent | 写代码 | 读写代码 |
| Test Agent | 写测试、执行测试 | 只读代码，可新增测试文件 |
| Review Agent | 审查质量/安全/边界 | 只读代码，只输出文本意见 |

### 9.2 执行铁律
- 每个阶段开头必须声明当前角色（如 [Role: Dev Agent]）
- Dev 不得执行 Test/Review 任务，反之亦然
- Review Agent 的审查结论是"完成"的唯一下闸条件
- 角色切换时输出分隔线 ==== 切换至 XXX ====
